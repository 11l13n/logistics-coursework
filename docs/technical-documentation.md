# Техническая документация проекта

## Назначение системы

Логистическая информационная система предназначена для автоматизации работы небольшой транспортной или логистической компании. Система помогает вести заявки на перевозку, хранить данные о клиентах, водителях и автомобилях, планировать маршруты, создавать путевые листы и получать отчеты по выполненным рейсам.

Основные пользователи системы:

- администратор;
- диспетчер;
- водитель.

Администратор управляет пользователями и справочниками. Диспетчер создает заявки и маршруты. Водитель работает с назначенными рейсами и путевыми листами.

## Состав проекта

Проект состоит из двух основных частей:

- `client` - клиентская часть на React;
- `server` - серверная часть на Node.js и Express.

Также в проекте есть:

- `server/prisma/schema.prisma` - описание структуры базы данных;
- `server/prisma/migrations` - миграции базы данных;
- `server/prisma/seed.js` - заполнение базы тестовыми данными;
- `server/src/swagger.js` - описание API для Swagger;
- `server/tests` - unit-тесты серверной логики.

## Архитектура приложения

Приложение построено по клиент-серверной архитектуре.

Клиентская часть отвечает за отображение интерфейса и отправку запросов к API. Пользователь работает с приложением через браузер. После входа в систему frontend сохраняет токен авторизации и использует его при обращении к защищенным маршрутам backend.

Серверная часть принимает HTTP-запросы, проверяет права доступа, выполняет бизнес-логику и обращается к базе данных через Prisma ORM. Данные хранятся в PostgreSQL.

Общая схема работы:

```text
Пользователь -> React frontend -> Express API -> Prisma ORM -> PostgreSQL
```

## База данных

Для хранения данных используется PostgreSQL. Структура базы описана в `server/prisma/schema.prisma`.

Основные сущности:

- `User` - пользователь системы;
- `Driver` - водитель;
- `Vehicle` - транспортное средство;
- `Client` - клиент компании;
- `CargoRequest` - заявка на перевозку;
- `CargoDeliveryPoint` - точки доставки по заявке;
- `Route` - маршрут;
- `Waypoint` - точки маршрута;
- `Waybill` - путевой лист.

Связи между сущностями:

- пользователь может быть связан с водителем;
- клиент имеет заявки на перевозку;
- заявка используется для создания маршрута;
- маршрут связан с водителем и автомобилем;
- маршрут имеет путевой лист;
- заявка и маршрут могут иметь несколько точек доставки.

## Авторизация и роли

Для авторизации используется JWT. После успешного входа сервер возвращает токен, который frontend передает в заголовке:

```text
Authorization: Bearer <token>
```

В системе используются роли:

- `ADMIN` - полный доступ к управлению пользователями и справочниками;
- `DISPATCHER` - работа с заявками, маршрутами, рекомендациями и отчетами;
- `DRIVER` - доступ только к своим маршрутам и путевым листам.

Проверка авторизации выполняется в `middleware/auth.js`, проверка ролей - в `middleware/roles.js`.

## Основная бизнес-логика

### Создание заявки

Диспетчер или администратор создает заявку на перевозку. В заявке указываются клиент, груз, вес, объем, адрес погрузки, адрес доставки и желаемая дата выполнения.

### Подбор водителя и автомобиля

Модуль рекомендаций проверяет:

- вес груза;
- грузоподъемность автомобиля;
- статус автомобиля;
- статус водителя;
- занятость ресурсов на выбранную дату.

Если подходящие водитель и автомобиль найдены, система возвращает рекомендацию и расчет планового расхода топлива.

### Создание маршрута

Маршрут создается на основе новой заявки. После создания маршрута:

- заявка получает статус `PLANNED`;
- водитель получает статус `BUSY`;
- автомобиль получает статус `BUSY`;
- создается путевой лист.

### Работа водителя

Водитель видит только свои маршруты. Он может начать рейс, завершить рейс и заполнить фактические данные путевого листа: пробег и расход топлива.

### Завершение рейса

При завершении рейса:

- маршрут получает статус `COMPLETED`;
- путевой лист получает статус `COMPLETED`;
- заявка получает статус `COMPLETED`;
- водитель и автомобиль снова становятся доступными.

## API-документация

API описано с помощью Swagger. После запуска backend документация доступна по адресу:

```text
http://127.0.0.1:4000/api-docs
```

OpenAPI JSON доступен по адресу:

```text
http://127.0.0.1:4000/api/openapi.json
```

Основные разделы API:

- `/api/auth`;
- `/api/users`;
- `/api/drivers`;
- `/api/vehicles`;
- `/api/clients`;
- `/api/cargo-requests`;
- `/api/routes`;
- `/api/recommendations`;
- `/api/waybills`;
- `/api/reports`.

## Переменные окружения

Для backend используется файл `server/.env`.

Пример:

```env
DATABASE_URL="postgresql://postgres:postgres@localhost:5432/logistics_coursework?schema=public"
JWT_SECRET="change-this-secret"
PORT=4000
HOST=127.0.0.1
CLIENT_URL="http://localhost:5173"
```

Для frontend используется файл `client/.env`.

Пример:

```env
VITE_API_URL="http://127.0.0.1:4000/api"
```

Файлы `.env` не добавляются в репозиторий, потому что в них могут храниться локальные настройки и секретные значения.

## Установка и запуск

Установка зависимостей:

```bash
npm install --prefix server
npm install --prefix client
```

Подготовка базы данных:

```bash
npm run prisma:generate
npm run prisma:migrate
npm run prisma:seed
```

Запуск backend:

```bash
npm run dev:server
```

Запуск frontend:

```bash
npm run dev:client
```

## Тестирование

Unit-тесты находятся в папке `server/tests`.

Запуск тестов:

```bash
npm test
```

Тесты проверяют:

- успешную авторизацию;
- отказ при неверном пароле;
- создание пользователя;
- проверку ролей;
- создание и редактирование заявки;
- подбор водителя и автомобиля;
- создание маршрута и путевого листа;
- сохранение фактических данных в путевом листе;
- начало и завершение рейса;
- формирование отчетов;
- доступ водителя только к своим маршрутам;
- вспомогательные функции для дат, адресов и маршрутов.

Тесты используют моковые данные и не требуют подключения к PostgreSQL.

## Сборка frontend

Проверка production-сборки:

```bash
npm run build:client
```

После сборки готовые файлы появляются в папке `client/dist`. Эта папка не хранится в репозитории, потому что ее можно получить заново командой сборки.
